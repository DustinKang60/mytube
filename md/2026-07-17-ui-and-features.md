**날짜**: 2026-07-17

## 1. 채널 목록 캐시 (커밋 `92f41a9`)

**문제**: 앱을 켜고 채널을 누를 때마다 매번 유튜브에서 목록을 다시 불러옴.
실제 목록은 하루~며칠에 한 번 바뀌는데 낭비.

**해결**: 채널별 영상 목록을 폰 `localStorage`에 저장.

```
채널 탭 → 캐시 있으면 즉시 표시(네트워크 요청 없음)
        → 캐시가 6시간 이내면 그대로 끝
        → 6시간 넘었으면 화면엔 캐시 보여주고, 뒤에서 조용히 갱신
```

- 갱신 실패해도 이미 보여준 캐시는 유지
- 채널 삭제 시 해당 캐시도 함께 삭제

**저장 위치는 폰(RAM 아닌 localStorage)이고 서버가 아님.** 이유:
서버는 항상 켜져 있지 않고, 서버에 둬도 폰이 매번 네트워크로 물어봐야 해서
"매번 다시 불러오는" 문제가 그대로 남기 때문.

## 2. 재생 모드 토글 (커밋 `c58cf2a`)

**요구**: 강연은 하나만 듣고 끝내고 싶고, 음악은 계속 듣고 싶다.

**구현**: 반복 토글 버튼 추가.

| 상태 | 동작 |
|---|---|
| 켜짐(선명) | **이어듣기** — 목록을 계속 재생, 마지막 곡 뒤엔 처음으로 순환 |
| 꺼짐(흐림) | **한 곡만** — 이 곡이 끝나면 정지 |

- `localStorage`에 저장, 기본값은 이어듣기(기존 동작 유지)
- **이전/다음 버튼은 모드와 무관하게 항상 동작** — 모드는 "곡이 저절로 끝났을 때"만 적용
- 두 재생 엔진(다운로드 Blob / 유튜브 임베드) 모두 같은 핸들러(`handleTrackEnded`)를 거침

## 3. 컨트롤 재구성 (커밋 `5e6988b`)

**문제**: 반복 버튼을 넣었더니 `⏮ ▶ ⏭ 🔁` 4개가 되면서 **좌우 대칭이 깨짐**.

**결정 (사용자와 상의 후)**: 이전/다음 버튼을 빼고 3개 대칭으로 재구성.

```
[1x]        ▶        [🔁]
배속       재생       반복
```

**이전/다음을 뺀 근거:**
- 바로 아래 목록에서 원하는 곡을 직접 탭하는 게 더 빠름
- 통째 다운로드 방식이라 이전/다음을 누를 때마다 **새로 ~25초 다운로드** 대기 → 장점이 사라짐
- **잠금화면·알림의 이전/다음은 그대로 유지됨** (MediaSession이 담당, 앱 화면 버튼과 별개)
  → `prevTrack`/`nextTrack` 함수는 그대로 남겨둠

**배속을 넣은 근거**: 구독 채널이 강연·경제·뉴스 위주라 배속 수요가 큼.
오디오가 이미 메모리에 있어 **즉시 적용**되고 음정도 유지됨.

- `1x → 1.25x → 1.5x → 2x → 1x` 순환, 1x가 아니면 강조 표시
- `load()`가 재생 속도를 초기화하므로 **곡 로드 후 다시 적용** 필요 (`applyPlaybackRate`)
- `localStorage`에 저장

**검증**: 좌우 간격 24px 동일, 재생 버튼 정중앙, 모바일 375px 오버플로 없음.

## 4. 브랜드 표기 (커밋 `8ffb8b3`)

로고 옆에 작은 글씨로 **작은앱공방** 추가. 로고와 경쟁하지 않도록
작고(0.7rem) 흐린 색(`--text-muted`)으로 처리.

## 개발 중 만난 함정

### 서비스워커 캐시 때문에 새 코드가 안 보임

이 앱의 서비스워커는 **캐시 우선**이라, 코드를 고쳐 배포해도 앱을 열면
**폰에 저장된 이전 코드**를 먼저 쓴다. 새 코드는 그다음 새로고침 때 로드된다.

실제로 이것 때문에 **"수정이 실패했다"고 오판할 뻔했다.**
v15를 올리고 테스트했는데 옛날 동작이 그대로여서, 처음엔 코드가 잘못된 줄 알았다.

**대응**:
- 에셋 바꿀 때마다 `sw.js`의 `CACHE_NAME` 버전을 반드시 올린다 (`mytube-vN`)
- 폰에서 확인할 땐: 앱 완전 종료(최근 앱에서 스와이프) → 다시 열기 → **당겨서 새로고침 2~3번**

### 프리뷰 탭에서 CSS 트랜지션이 얼어붙음

브라우저 프리뷰 탭이 백그라운드면 **CSS 트랜지션이 진행되지 않아**,
클릭 직후 계산된 스타일을 읽으면 **이전 상태 값이 그대로 나온다.**

반복 버튼의 켜짐/꺼짐 스타일이 안 바뀌는 것처럼 보여서 한참 헤맸는데,
실제로는 정상이었고 트랜지션이 얼어붙은 것뿐이었다.

**대응**: 검증할 때 트랜지션을 끄고 목표 스타일을 읽는다.

```js
const kill = document.createElement('style');
kill.textContent = '*{transition:none !important}';
document.head.appendChild(kill);
// ... 여기서 getComputedStyle로 읽기
kill.remove();
```

### 콘솔의 `YT Player error: 150`

이전 세션의 잔여 로그였다. 실시간 측정 시 5초간 **0건**, `nextTrack` 호출도 0건으로
재생에 영향 없음을 확인했다. 새 에러처럼 보여도 실제로 발생 중인지 확인이 필요하다.
