# 화면 끄면 2분 만에 끊기던 문제 — 진단과 해결

**날짜**: 2026-07-17
**결과**: 서버 다운로드 속도 **30 KB/s → 5.5 MB/s (약 180배)**. 1시간짜리 클립(64.9MB)이 12초에 받아짐.

---

## 1. 증상

- 앱에서 곡을 재생하면 광고 없이 잘 나옴
- 그런데 **4~5분 후(실제로는 약 2분) 오디오가 끊김**
- 특히 **화면을 끄면** 재현됨

## 2. 진단 과정 (막다른 길 포함)

### 2-1. 첫 가설: "Cloudflare 터널이 스트림을 끊는다" — ❌ 틀렸음

서버 로그에 이런 게 반복적으로 찍혔다:

```
GET /audio/YyWetcTtMa8 206
ERR error="stream 517 canceled by remote with error code 0"
    dest=https://xxx.trycloudflare.com/audio/YyWetcTtMa8 ip=198.41.192.67
```

`ip=198.41.192.67`은 Cloudflare 엣지 서버라서, **무료 Quick Tunnel이 장시간 연결을 끊는 것**으로 판단했다.
→ 이 가설을 믿고 **클라이언트 자동 재연결**을 구현했다. (커밋 `fb4ca9b`)

### 2-2. 재연결 로직의 자체 버그 — 광고 발생

재연결을 넣자 **없던 광고가 나오기 시작**했다.

원인: 워치독이 2초마다 도는데, 스트림을 다시 불러오는 몇 초 동안에도
"재생 위치가 안 움직인다"며 재연결을 계속 다시 호출 → 재시도 한도(5회)를
**10초 만에 소진** → 유튜브 임베드(광고 있음)로 전환.

수정: 재연결 진행 중엔 워치독이 개입하지 않도록 잠금(`serverReconnecting`) 추가. (커밋 `97b33f4`)

### 2-3. 화면 끄면 재연결이 아예 안 됨 — 타이머가 얼어붙음

로그 분석 결과:

```
23:35:00Z stream 517 canceled → GET /audio 206  (재연결 성공, 화면 켜져 있던 시점)
23:37:11Z stream 521 canceled → (재연결 요청 없음)  ← 화면 끈 뒤, 여기서 끊김
```

원인: 재연결 감지가 `setInterval` 타이머 기반인데,
**모바일 브라우저는 화면이 꺼지면 타이머를 얼려버린다.**

→ 타이머 대신 **이벤트 기반**(`stalled`/`error`/`ended`)으로 변경. (커밋 `c7cbd97`)

### 2-4. 이벤트 기반으로도 실패 — 접근 자체가 틀렸음

v15를 로드하고 화면 끈 채 테스트했으나 **여전히 5분을 못 채우고 끊김**.

여기서 방향 전환: **증상(끊긴 뒤 다시 잇기)이 아니라 원인을 봐야 한다.**

### 2-5. 결정적 측정 — 터널은 범인이 아니었다

브라우저를 빼고 `curl`로 직접 측정했다.

| 경로 | 속도 | 결과 |
|---|---|---|
| 터널 경유 (PC → Cloudflare → 서버) | ~31 KB/s | 200초 내내 **안 끊김** (내가 건 제한에 걸림, exit=28) |
| localhost 직접 (터널 우회) | ~28 KB/s | 같은 속도 |
| **병렬 4연결 (localhost)** | **각 ~30 KB/s = 합 122 KB/s** | 나눠 갖지 않음 |

**해석:**
1. 터널 경유와 localhost 속도가 **같다** → 터널은 범인이 아니다.
   → **ngrok 등으로 터널을 바꿔도 소용없다.** (다른 세션의 Claude가 이 점은 맞았다)
2. `curl`은 200초를 끊김 없이 받았다 → 터널은 장시간 연결을 잘 유지한다.
   로그의 "stream canceled"는 터널이 끊은 게 아니라 **브라우저가 버퍼 채우고 연결을 닫은 것**이었다.
3. **~30 KB/s는 사실상 실시간 재생 속도와 같다** = googlevideo가 연결당 실시간 배속으로만 흘려보냄(pacing).
4. **pacing은 "연결당"** 이다 → 병렬로 나눠 받으면 우회된다. ← **해결의 열쇠**

### 2-6. 진짜 원인

```
googlevideo가 연결당 ~30KB/s(실시간)로만 전송
        ↓
단일 연결 스트리밍은 재생보다 앞서 나갈 수 없음 (버퍼를 못 쌓음)
        ↓
화면 끄면 브라우저가 백그라운드 네트워크를 중단
        ↓
쌓아둔 약 2분치 버퍼를 다 쓰면 재생 정지
```

클라이언트 재연결로는 절대 못 이긴다 (백그라운드에서 타이머·네트워크가 얼어붙으므로).

## 3. 해결책 (2단계)

### 3-1. 서버: 병렬 다운로드 + 디스크 캐시 (`server/mytube-server.py`)

`GET /audio/<id>` 요청 시:
1. 디스크 캐시에 있으면 → 로컬 파일 바로 서빙 (Range 지원)
2. 없으면 → **16개 병렬 range 요청**으로 통째 다운로드 → 디스크 저장 → 서빙

```python
DL_CONNS = 16          # 병렬 연결 수 (MYTUBE_DL_CONNS로 조절)
CACHE_MAX_BYTES = 4GB  # 초과 시 오래된 것부터 삭제(LRU)
```

- `os.pwrite`로 각 스레드가 자기 오프셋에 직접 기록 (스레드 안전)
- `.part` 임시 파일 → 완료 시 `os.replace`로 원자적 교체
- 실패 시 기존 단일 스트리밍으로 폴백

### 3-2. 앱: 통째로 받아서 메모리에서 재생 (`app.js`)

```
곡 탭 → fetch로 파일 전체를 메모리에 다운로드(진행률 %) → Blob URL로 재생
```

메모리에 전부 있으니 **네트워크가 더 이상 필요 없음** → 화면 꺼도 끝까지 재생.
(다운받은 팟캐스트가 백그라운드에서 잘 나오는 것과 같은 원리)

- 새 곡 탭 시: 이전 다운로드 `abort` + 이전 곡 즉시 정지 + 이전 Blob 메모리 해제
- 메모리엔 **항상 현재 곡 1개만** 유지 (다음 곡 가면 이전 건 삭제)
- 쓸모없어진 재연결/워치독 코드는 전부 제거

## 4. 실측 결과

배포 후 서버 로그:

```
[cache] Avjs7n1TWOw: 16.9MB in  3s (5.48 MB/s, 16 conns)
[cache] Wm-bcbLLQwk: 48.8MB in  9s (5.24 MB/s, 16 conns)
[cache] Yd4Oc_Nxe8w: 64.9MB in 12s (5.47 MB/s, 16 conns)
```

**30 KB/s → 5.5 MB/s (약 180배).** 1시간짜리 클래식이 12초.

## 5. 배운 것

1. **로그의 표면만 보면 틀린다.** "stream canceled by remote"를 보고 터널을 범인으로 지목했지만,
   실제론 브라우저가 닫은 것이었다. `curl`로 브라우저를 빼고 측정하니 진실이 드러났다.
2. **증상이 아니라 원인을 고쳐야 한다.** 재연결 로직을 3번이나 고쳤지만 전부 실패했다.
   원인(pacing)을 없애자 재연결 자체가 필요 없어졌다.
3. **측정 없이 추측하면 안 된다.** 병렬 4연결 테스트 한 번이 해결의 열쇠였다.
4. **같은 도구라도 문제에 따라 답이 다르다.** ngrok은 "끊김" 해결엔 무의미하지만,
   "주소 고정"엔 정확한 도구다. 헷갈리면 안 된다.

## 6. 남은 것 / 한계

- **최종 확인 미완**: 화면 끄고 5분 이상 재생 유지되는지 실사용 검증 필요
- **첫 재생 대기**: 캐시 안 된 곡은 서버 다운로드(수 초~십수 초) 대기. 캐시된 뒤엔 즉시
- **곡 넘김(이어듣기)**: 화면 끈 채 다음 곡으로 넘어갈 때 새 다운로드가 필요해서,
  브라우저가 백그라운드 요청을 지연시키면 안 될 수 있음 → 필요 시 **다음 곡 미리 받기(prefetch)** 로 해결 가능
- **라이브 스트림**: yt-dlp가 m4a를 못 뽑아 재생 불가 (지원 안 하기로 결정)
